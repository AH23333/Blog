#### 1. HTTP/1.1 完整请求-响应（含长连接）
```mermaid
sequenceDiagram
    participant Client as 客户端浏览器
    participant Server as Web服务器(TCP 80)

    Note over Client,Server: ===== 应用层 · HTTP/1.1 长连接流程 =====
    Note left of Client: 解析域名得到服务器IP
    
    %% TCP建连
    Client->>+Server: TCP三次握手建立连接
    Note over Client,Server: TCP连接建立，默认长连接 Connection: keep-alive

    %% 第一个请求
    Client->>Server: GET /index.html HTTP/1.1<br/>Host: www.example.com<br/>Connection: keep-alive<br/>Cookie: session=xxx
    Note right of Server: 解析请求行+请求头<br/>查找本地资源文件
    
    Server-->>Client: HTTP/1.1 200 OK<br/>Content-Type: text/html<br/>Content-Length: 1024<br/>Connection: keep-alive<br/><br/>[HTML页面内容]
    Note left of Client: 渲染HTML，发现内嵌CSS/图片资源

    %% 第二个请求（同一条TCP连接，长连接复用）
    Client->>Server: GET /style.css HTTP/1.1<br/>Host: www.example.com<br/>Connection: keep-alive
    Server-->>Client: HTTP/1.1 200 OK<br/>Content-Type: text/css<br/>Content-Length: 512<br/><br/>[CSS样式内容]

    Note over Client,Server: 长连接保持，可继续复用TCP连接<br/>超时无数据则服务器主动关闭连接
```

#### 2. DNS 域名解析（递归+迭代完整流程）
```mermaid
sequenceDiagram
    participant Host as 用户主机
    participant LocalDNS as 本地DNS服务器
    participant RootDNS as 根域名服务器
    participant TLD as .com顶级域服务器
    participant AuthDNS as example.com权威服务器

    Note over Host,AuthDNS: ===== 应用层 · DNS完整解析流程 =====
    Note over Host: 浏览器输入www.example.com<br/>检查浏览器缓存→系统缓存→hosts文件→无结果

    %% 递归查询
    Host->>+LocalDNS: 递归查询请求 (UDP 53)<br/>Q: www.example.com 的A记录
    Note left of LocalDNS: 递归查询：必须返回最终IP，不能返回下一跳

    %% 迭代查询1
    LocalDNS->>+RootDNS: 迭代查询请求 (UDP 53)<br/>Q: www.example.com 的A记录
    RootDNS-->>-LocalDNS: 迭代响应<br/>返回 .com 顶级域服务器地址列表
    Note right of RootDNS: 迭代查询：只返回下一级服务器地址，不代查

    %% 迭代查询2
    LocalDNS->>+TLD: 迭代查询请求 (UDP 53)<br/>Q: www.example.com 的A记录
    TLD-->>-LocalDNS: 迭代响应<br/>返回 example.com 权威服务器地址

    %% 迭代查询3
    LocalDNS->>+AuthDNS: 迭代查询请求 (UDP 53)<br/>Q: www.example.com 的A记录
    AuthDNS-->>-LocalDNS: 迭代响应<br/>A记录: www.example.com → 202.103.0.10<br/>TTL = 3600s

    %% 返回最终结果
    LocalDNS-->>-Host: 递归查询响应<br/>www.example.com → 202.103.0.10
    Note over Host,LocalDNS: 双方均写入本地DNS缓存，TTL有效期内直接复用
```

#### 3. DHCP 动态地址分配（四阶段完整流程）
```mermaid
sequenceDiagram
    participant Client as DHCP客户端(0.0.0.0)
    participant Server as DHCP服务器(192.168.1.1)

    Note over Client,Server: ===== 应用层 · DHCP四阶段流程 =====
    Note left of Client: 主机启动，无IP地址<br/>UDP源端口68，目的端口67

    %% 阶段1：发现
    Client->>Server: 广播 DHCP Discover<br/>源IP:0.0.0.0 目的IP:255.255.255.255<br/>源MAC:MAC_C 目的MAC:FF:FF:FF:FF:FF:FF<br/>请求：申请IP地址
    Note over Client,Server: 第一阶段：发现阶段，全网广播寻找DHCP服务器

    %% 阶段2：提供
    Server-->>Client: 单播/广播 DHCP Offer<br/>分配IP:192.168.1.100<br/>子网掩码:255.255.255.0<br/>网关:192.168.1.1 DNS:8.8.8.8<br/>租期:86400s
    Note over Client,Server: 第二阶段：提供阶段，服务器返回预分配地址

    %% 阶段3：请求
    Client->>Server: 广播 DHCP Request<br/>源IP:0.0.0.0 目的IP:255.255.255.255<br/>确认接受:192.168.1.100
    Note over Client,Server: 第三阶段：请求阶段，广播确认接受该地址（通知其他服务器）

    %% 阶段4：确认
    Server-->>Client: 单播/广播 DHCP ACK<br/>正式确认分配192.168.1.100<br/>租期、网关、DNS等参数生效
    Note over Client,Server: 第四阶段：确认阶段，地址正式生效，客户端配置完成

    Note left of Client: 租期过半时自动发送DHCP Request续租<br/>租期到期未收到ACK则释放地址
```

### （二）传输层协议时序图
#### 1. TCP 三次握手（带完整状态与序号）
```mermaid
sequenceDiagram
    participant Client as 客户端
    participant Server as 服务器

    Note over Client,Server: ===== 传输层 · TCP三次握手 =====
    Note left of Client: 状态: CLOSED<br/>随机初始序号 seq = x
    Note right of Server: 状态: LISTEN<br/>随机初始序号 seq = y

    %% 第一次握手
    Client->>+Server: SYN = 1, ACK = 0<br/>seq = x<br/>窗口: win = 4096<br/>MSS = 1460
    Note left of Client: 状态: SYN-SENT
    Note right of Server: 状态: SYN-RCVD<br/>SYN报文不携带数据，消耗1个序号

    %% 第二次握手
    Server-->>-Client: SYN = 1, ACK = 1<br/>seq = y, ack = x + 1<br/>窗口: win = 2048
    Note left of Client: 收到SYN+ACK<br/>状态: ESTABLISHED

    %% 第三次握手
    Client->>+Server: SYN = 0, ACK = 1<br/>seq = x + 1, ack = y + 1<br/>可携带应用层数据
    Note right of Server: 收到ACK<br/>状态: ESTABLISHED

    Note over Client,Server: 连接全双工建立，双向可传输数据<br/>第三次握手可携带数据，不携带则不消耗序号
```

#### 2. TCP 四次挥手（带完整状态与2MSL）
```mermaid
sequenceDiagram
    participant Client as 主动关闭方
    participant Server as 被动关闭方

    Note over Client,Server: ===== 传输层 · TCP四次挥手 =====
    Note over Client,Server: 初始状态: 双方均为 ESTABLISHED

    %% 第一次挥手
    Client->>+Server: FIN = 1, ACK = 1<br/>seq = u, ack = v
    Note left of Client: 状态: FIN-WAIT-1<br/>FIN报文不携带数据，消耗1个序号
    Note right of Server: 收到FIN<br/>状态: CLOSE-WAIT

    %% 第二次挥手
    Server-->>-Client: ACK = 1<br/>seq = v, ack = u + 1
    Note left of Client: 收到ACK<br/>状态: FIN-WAIT-2
    Note over Server: 半关闭状态：客户端不再发数据<br/>服务器仍可发送剩余数据

    %% 服务器发送完剩余数据
    Server->>Client: [剩余应用数据传输]

    %% 第三次挥手
    Server->>+Client: FIN = 1, ACK = 1<br/>seq = w, ack = u + 1
    Note right of Server: 数据发送完毕<br/>状态: LAST-ACK

    %% 第四次挥手
    Client-->>-Server: ACK = 1<br/>seq = u + 1, ack = w + 1
    Note left of Client: 状态: TIME-WAIT<br/>等待 2*MSL (最长报文寿命)
    Note right of Server: 收到ACK<br/>状态: CLOSED

    Note over Client: 2MSL计时结束<br/>状态: CLOSED<br/>1. 保证最后一个ACK可达，丢失可重传<br/>2. 清除网络中残余报文，避免污染新连接
```

#### 3. TCP 停止等待协议（可靠传输基础）
```mermaid
sequenceDiagram
    participant S as 发送方
    participant R as 接收方

    Note over S,R: ===== 传输层 · 停止等待可靠传输 =====
    
    %% 正常传输
    Note over S,R: --- 场景1：正常传输 ---
    S->>R: 报文段 M1 (seq=1)
    Note over S: 启动超时计时器，等待ACK
    R-->>S: ACK1 (确认号=2)
    Note over S: 收到ACK，停止计时器<br/>发送下一个报文

    %% 报文丢失
    Note over S,R: --- 场景2：报文丢失，超时重传 ---
    S->>R: 报文段 M2 (seq=2) [传输中丢失]
    Note over S: 启动超时计时器
    Note over S: 超时时间到，未收到ACK
    S->>R: 重传报文段 M2 (seq=2)
    R-->>S: ACK2 (确认号=3)
    Note over S: 收到ACK，停止计时器

    %% ACK丢失
    Note over S,R: --- 场景3：ACK丢失，超时重传 ---
    S->>R: 报文段 M3 (seq=3)
    R-->>S: ACK3 (确认号=4) [传输中丢失]
    Note over S: 超时时间到，未收到ACK
    S->>R: 重传报文段 M3 (seq=3)
    Note over R: 收到重复报文，丢弃<br/>重新发送ACK3
    R-->>S: ACK3 (确认号=4)
    Note over S: 收到ACK，停止计时器
```

#### 4. TCP 滑动窗口与流量控制
```mermaid
sequenceDiagram
    participant S as 发送方
    participant R as 接收方

    Note over S,R: ===== 传输层 · 滑动窗口与流量控制 =====
    Note over S: 初始发送窗口 swnd = min(cwnd, rwnd) = 5 MSS
    Note over R: 初始接收窗口 rwnd = 5 MSS

    %% 第一批发送
    S->>R: 报文1 (seq=1)
    S->>R: 报文2 (seq=2)
    S->>R: 报文3 (seq=3)
    Note over S: 已发送未确认：3个<br/>窗口剩余可发送：2个

    R-->>S: ACK=4, rwnd=3 (累计确认1~3，接收窗口缩为3)
    Note over S: 窗口滑动，已发送未确认清零<br/>发送窗口更新为3 MSS

    %% 第二批发送
    S->>R: 报文4 (seq=4)
    S->>R: 报文5 (seq=5)
    S->>R: 报文6 (seq=6)
    Note over S: 窗口已满，停止发送，等待ACK

    R-->>S: ACK=7, rwnd=0 (接收方缓存满，通知零窗口)
    Note over S: 发送窗口变为0，停止发送<br/>启动持续计时器

    Note over R: 接收方处理数据，缓存释放
    R-->>S: 窗口更新报文 ACK=7, rwnd=4
    Note over S: 收到窗口更新，恢复发送<br/>发送窗口更新为4 MSS

    S->>R: 报文7 (seq=7)
```

#### 5. TCP 快重传 + 快恢复详细流程
```mermaid
sequenceDiagram
    participant S as 发送方
    participant R as 接收方

    Note over S,R: ===== 传输层 · 快重传与快恢复 =====
    Note over S: 当前 cwnd = 20 MSS, ssthresh = 16 MSS<br/>处于拥塞避免阶段

    %% 批量发送，中间丢包
    S->>R: 报文1
    S->>R: 报文2
    S->>R: 报文3 [丢失]
    S->>R: 报文4
    S->>R: 报文5
    S->>R: 报文6

    %% 累计确认+重复ACK
    R-->>S: ACK=3 (累计确认1~2，期望3)
    R-->>S: ACK=3 (第1个重复ACK，收到报文4)
    R-->>S: ACK=3 (第2个重复ACK，收到报文5)
    R-->>S: ACK=3 (第3个重复ACK，收到报文6)

    Note over S: 连续收到3个重复ACK<br/>触发快重传：立即重传丢失报文，不等超时
    S->>R: 重传报文3

    Note over S: 触发快恢复：<br/>ssthresh = cwnd/2 = 10 MSS<br/>cwnd = 新 ssthresh = 10 MSS<br/>直接进入拥塞避免阶段

    R-->>S: ACK=7 (累计确认1~6全部收到)
    Note over S: cwnd = 11 MSS (拥塞避免，每RTT+1)

    Note over S,R: 对比超时重传：超时会将cwnd重置为1，重新慢开始<br/>快恢复仅减半窗口，避免速率骤降
```

### （五）物理层说明
物理层不存在对等实体之间的协议交互时序，它定义的是**硬件传输的电气、机械、功能、规程四大特性**，核心是将数据链路层的比特流转换为物理信号传输，不涉及逻辑交互。
- 核心考点：奈奎斯特定理、香农定理、数字编码方式、模拟调制方式、信道复用技术、传输介质分类。
